# Ingredient Recommendation System - food.com
---

## Project Overview

food.com hired us as consultants to build a personalised **ingredient recommendation system** on top of their recipe and user-review dataset. The system has three core capabilities:

1. **Top-N Ingredient Recommendation** — Given a user's past recipe reviews, recommend the top-N ingredients they are most likely to want next, ranked by relevance.
2. **Recipe Matching** — Given a list of ingredients (e.g. the personalised Top-N), return the best-matching recipes from the catalogue.
3. **Ingredient Substitution** — Given a specific ingredient, suggest the most relevant culinary alternatives using co-occurrence profile similarity.

Two optional objectives were also addressed:

- **Coverage & Diversity** — How broadly the system explores the 14,942-ingredient catalogue and how varied each user's recommendation list is.
- **Calorie-aware Re-ranking** — Incorporating nutritional data to promote healthier recommendations.

## Key Results

| Metric | Value |
|--------|-------|
| Precision@5 (test set) | **0.58** |
| Precision@10 (test set) | **0.50** |
| Recall@10 (test set) | **0.12** |
| F1@10 (test set) | **0.19** |
| Catalogue coverage | 0.27% |
| Avg within-list diversity | 0.25 (1 − cosine) |
| Calorie reduction from re-ranking | −20 kcal avg |

The model achieves ~50% precision at K=10 in a fully cold-start evaluation setting (test users were never seen during training), which is strong for a co-occurrence-based approach on a large, sparse item space.

## How to Run

### 1. Clone the repository

```bash
git clone https://github.com/<your-username>/ingredient-recommendation.git
cd ingredient-recommendation
```

### 2. Set up the Python environment

The project uses a Conda environment with Python 3.10:

```bash
conda create -n rec_system python=3.10
conda activate rec_system
pip install -r requirements.txt
```

### 3. Add the data files

Download `metadata.csv` and `train.csv` from the course data portal and place them in:

```
data/metadata.csv
data/train.csv
```

### 4. Run the notebook

Launch Jupyter and open the notebook **from the project root** so that relative paths (`data/`, `outputs/`) resolve correctly:

```bash
cd ingredient-recommendation/   # project root
jupyter notebook notebook/group_submission.ipynb
```

Run all cells top to bottom (Kernel → Restart & Run All). Charts are saved automatically to `outputs/charts/` and processed data splits to `data/`.

## Methods

The system is built entirely on **ingredient co-occurrence**, a content-based approach demonstrated in the course curriculum:

- A co-occurrence matrix is built from all 231,637 recipes: `cooccurrence[a][b]` = number of recipes where ingredients `a` and `b` both appear.
- **Recommendation** scores unseen ingredients by aggregating co-occurrence counts with the user's known ingredient history, with a global-popularity fallback for sparse cases.
- **Substitution** represents each ingredient as its co-occurrence profile vector and ranks candidates by cosine similarity to the query ingredient.
- **Recipe matching** scores recipes by overlap count and Jaccard similarity against a query ingredient list.

---

## Evaluation Protocol

This is a **Top-N ranking problem**, not a rating-prediction problem. Accordingly, RMSE and MAE are not used. Evaluation follows a cold-start, held-out protocol:

1. Users are split 70/15/15 at the user level — all validation and test users are unseen during training.
2. For each held-out user, their ingredient list is split 50/50: the first half acts as the input "seed"; the second half is the ground truth.
3. Precision@K and Recall@K are computed against the held-out half.

---

## Limitations

- **Coverage bias**: only ~0.27% of the 14,942-ingredient catalogue is ever recommended — the model is dominated by common pantry staples.
- **Cold-start users**: recommendations rely entirely on population-level co-occurrence patterns rather than individual history, making personalisation limited for users with unusual tastes.
- **Substitution ambiguity**: the co-occurrence profile method cannot distinguish complements (butter + sugar) from true substitutes; adding ingredient-category metadata would improve this.
- **Calorie proxy**: the calorie score for an ingredient is the average calorie content of the recipes it appears in, not the ingredient's own caloric value.

## Future Work

- Implement matrix factorisation (e.g. SVD, ALS) or neural collaborative filtering for richer latent representations.
- Add diversity-forcing mechanisms (e.g. Maximum Marginal Relevance) to improve catalogue coverage.
- Use ingredient category metadata (fats, proteins, starches) to produce more meaningful substitutions.
- Extend the calorie-aware objective to a multi-nutrient health score (protein, sodium, sugar, fat).
- Explore time-aware evaluation (temporal train/test splits) for a more realistic deployment simulation.
